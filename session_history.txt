  1. SAFETY LOGIC AUDIT — Strict vs Drift

  I traced every code path through normalize_batch. Here's what I found:

  All paths accounted for (no files fall through):
  ┌──────────────────────┬─────────────────────────┬─────────────────────────┐
  │      Condition       │       Strict Mode       │       Drift Mode        │
  ├──────────────────────┼─────────────────────────┼─────────────────────────┤
  │ Silent/too quiet     │ SKIPPED                 │ SKIPPED                 │
  ├──────────────────────┼─────────────────────────┼─────────────────────────┤
  │ Peak exceeds ceiling │ NEEDS_LIMITING (copied) │ Gain reduced, processed │
  ├──────────────────────┼─────────────────────────┼─────────────────────────┤
  │ Peak OK              │ Processed normally      │ Processed normally      │
  ├──────────────────────┼─────────────────────────┼─────────────────────────┤
  │ Upsample blocked     │ BLOCKED                 │ BLOCKED                 │
  ├──────────────────────┼─────────────────────────┼─────────────────────────┤
  │ Exception            │ FAILED                  │ FAILED                  │
  └──────────────────────┴─────────────────────────┴─────────────────────────┘
  BUG FOUND — LUFS meter uses wrong sample rate after resampling

  normalize_lufs_core.py:381 creates the meter at the original rate:
  meter = pyln.Meter(rate)

  But after resampling at line 480-483, the data is at output_rate. Then at
  lines 522-525, final_lufs is measured with the original-rate meter against
  resampled data:
  final_lufs = meter.integrated_loudness(normalized_data)  # data is at 
  output_rate, meter is at rate

  pyloudnorm.Meter bakes the sample rate into its filter coefficients. If a
  96kHz file is downsampled to 48kHz, the final LUFS measurement is wrong. Fix: 
  recreate the meter after resampling or conditionally create a new one:
  final_meter = pyln.Meter(output_rate) if output_rate != rate else meter

  Edge case in Drift mode — original already exceeds ceiling

  At normalize_lufs_core.py:439-442:
  original_peak = measure_true_peak(data, rate)
  headroom = peak_ceiling - original_peak  # NEGATIVE if original already hot
  max_safe_gain_db = headroom
  actual_gain_db = min(gain_db, max_safe_gain_db)

  If the source file already exceeds the peak ceiling (e.g., original peak =
  -0.5 dBTP, ceiling = -1.0 dBTP), then headroom = -0.5 and the file gets
  attenuated. This is technically correct (it protects the peak), but the user
  gets no specific warning that the file was made quieter than the original.
  Consider logging this as a distinct status like ATTENUATED.

  Dither + clip interaction — safe in practice

  Line 514 applies dither, then line 517 clips to [-1.0, 1.0]. TPDF dither at
  16-bit adds ~±0.00003 (~-90 dB). Since peak ceilings are typically -1.0 dBTP
  or lower, there's >1 dB of headroom before 0 dBFS. No real risk here.

  needs_limiting path robustness

  Line 414: needs_limiting_path.mkdir(exist_ok=True) — this works because the
  parent (batch_path) is already created by normalized_path.mkdir(parents=True) 
  on line 335. Cross-platform safe. No issue.

  Missing reason field in general exceptions

  At line 575-579, the FAILED error dict has no reason key, unlike all other 
  entries. If downstream code iterates results expecting a uniform schema, this
  will break. Add 'reason': 'exception'.

  2. ARCHITECTURE & PERFORMANCE

  Main loop analysis for 1000+ files

  The bottleneck is measure_true_peak — it 4x oversamples the entire file via
  SOXR, then scans for max. This is called:
  - Once per normal file (line 407, on test data)
  - Twice per drift-mode peak-exceeded file (line 407 on test data + line 439 on
   original)

  The second call is avoidable. You can derive the original peak mathematically:
  # Instead of measuring original peak separately:
  # predicted_peak = original_peak + gain_db (in dB domain)
  # So: original_peak = predicted_peak - gain_db
  original_peak_estimated = predicted_peak - gain_db

  This eliminates one full True Peak measurement per drift-mode file. For a
  batch where 30% of files hit the ceiling, that's a meaningful speedup.

  import soxr inside the loop (lines 214, 479) — Python caches modules after
  first import, so this is a dictionary lookup, not a re-initialization. No
  issue.

  Multiprocessing feasibility

  The per-file processing is embarrassingly parallel, but there are
  complications:
  Concern: progress_callback
  Issue: GUI callback, not serializable
  Solution: Main thread collects futures, updates progress
  ────────────────────────────────────────
  Concern: result_callback
  Issue: Same
  Solution: Same
  ────────────────────────────────────────
  Concern: Logging
  Issue: File handler shared across processes
  Solution: Each worker returns results; main thread logs
  ────────────────────────────────────────
  Concern: CSV writing
  Issue: Sequential by nature
  Solution: Collect all results, write once at end (already the case)
  ────────────────────────────────────────
  Concern: stop_requested
  Issue: Shared state
  Solution: Use multiprocessing.Event
  Recommended pattern for v3.0:
  from concurrent.futures import ProcessPoolExecutor, as_completed

  def process_single_file(audio_path, target_lufs, peak_ceiling, ...):
      """Standalone function — no self, no callbacks, fully serializable."""
      # ... all processing logic ...
      return result_dict

  # In normalize_batch:
  with ProcessPoolExecutor(max_workers=os.cpu_count()) as pool:
      futures = {pool.submit(process_single_file, f, ...): f for f in
  audio_files}
      for future in as_completed(futures):
          result = future.result()
          self.results.append(result)
          if self.progress_callback:
              self.progress_callback(...)

  This keeps the GUI responsive, parallelizes the heavy DSP, and centralizes all
   I/O in the main thread. Expect 3-6x speedup on modern machines for 1000+ file
   batches (bottleneck shifts to disk I/O).

  Modern UI: Modern

  Drag & Drop: Web-native
  PyInstaller: Not applicable
  Audio App Fit: Wrong fit
  My recommendation: PySide6 for v3.0

  Reasons specific to your use case:

  1. Real-time loudness metering — Qt has QGraphicsView and QPainter for drawing
   VU meters, loudness histograms, and waveform displays. CustomTkinter has no
  equivalent without embedding matplotlib (heavy, slow).
  2. Drag-and-drop — Qt handles native file drag-and-drop out of the box
  (QDragEnterEvent). CustomTkinter requires the third-party tkdnd extension
  which is fragile on Windows.
  3. Table views for batch results — QTableView with QAbstractTableModel gives 
  you sortable, scrollable batch result tables. Your current log window is a
  text box; a proper table would let users sort by status, filter failures, etc.
  4. Thread-safe signals — Qt's signal/slot system is thread-safe by design.
  Your current root.after(0, lambda: ...) pattern works but is fragile. Qt
  signals can cross thread boundaries natively.
  5. PyInstaller compatibility — PySide6 produces larger executables (~80MB vs
  ~30MB) but is well-supported. Use --exclude-module flags to strip unused Qt
  modules (WebEngine, 3D, etc.) to bring it down to ~45MB.

  Migration cost: Medium. Your GUI is ~800 lines. A PySide6 rewrite would be
  ~600-700 lines (Qt's layout system is more concise). The core engine needs
  zero changes.

  If you want to stay lightweight: Keep CustomTkinter. It's working, it ships
  small, and your v2.x users won't notice a difference. Only move to PySide6 if
  you're adding real-time visualization (meters, waveforms) — that's where
  CustomTkinter truly can't compete.

  4. v3.0 KILLER FEATURES — Game Audio Focus

  Feature 1: LRA (Loudness Range) Analysis + Gating

  What: Measure and report EBU R128 Loudness Range (LRA) for every file.
  Optionally filter/flag files outside an LRA threshold (e.g., ASWG recommends
  LRA < 20 LU for game dialogue).

  Why it matters: Game audio middleware (Wwise, FMOD) expects consistent dynamic
   range. A gunshot SFX with LRA of 25 LU next to dialogue at LRA 8 LU creates
  mixing nightmares. Studios currently use Nugen or iZotope for this — having it
   in a batch tool is a competitive advantage.

  Implementation: pyloudnorm doesn't expose LRA directly, but you can compute it
   from the gated short-term loudness distribution per EBU R128 s1.
  Alternatively, use the loudness package or implement the algorithm
  (percentile-based on short-term LUFS blocks).

  CSV output: Add lra_lu column + optional lra_pass boolean.

  Feature 2: BWF/iXML Metadata Injection

  What: Write standardized metadata into output WAV files: originator,
  description, loudness value, coding history, and iXML tags for Wwise/FMOD
  import.

  Why it matters: Game audio pipelines require metadata for asset management. 
  Currently, after normalizing, engineers manually tag files or use a separate
  tool. Embedding the LUFS value, target standard, and processing chain directly
   into BWF chunks means Wwise can auto-read loudness and the DAM (digital asset
   manager) can search/filter.

  Implementation: Use soundfile to write WAV files (already doing this), then 
  use the pysoundfile low-level API or a library like wavinfo to inject BEXT 
  (Broadcast Extension) and iXML chunks. Key fields:
  - LoudnessValue (BEXT)
  - LoudnessRange (BEXT)
  - MaxTruePeakLevel (BEXT)
  - iXML: custom tags for ASWG standard, preset name, processing date

  Feature 3: Watch Folder with Profile Presets

  What: A "hot folder" mode where the normalizer monitors a directory and 
  auto-processes new files using a saved profile (preset + settings). Output
  goes to a configured destination. Multiple watch folders with different
  profiles.

  Why it matters: Game audio teams generate hundreds of SFX per day across
  multiple designers. A watch folder per deliverable target (console build,
  mobile build, trailer mix) eliminates manual batch runs entirely. This is how
  professional facilities work (SADiE, Emotion Systems) — bringing that workflow
   to indie/mid-size studios.

  Implementation: Use watchdog library for filesystem monitoring. Each watch
  profile stores: input path, output path, preset, bit depth, sample rate,
  strict/drift mode. Run as a background service or a minimized tray icon
  (PySide6 has QSystemTrayIcon).
